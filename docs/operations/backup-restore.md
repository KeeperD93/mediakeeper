# Backup restore runbook

> **Audience.** This runbook is the operator-facing companion to the
> in-code restore primitives (`backend/services/backup/restore.py` and
> `POST /api/backup/restore*` endpoints). It documents the **manual**
> end-to-end recovery procedure that you have to run yourself in a real
> incident, because the application code does **not** replay
> `pg_dump.sql` automatically.
>
> **Scope.** Restoring an instance from a `mediakeeper_backup_*.zip`
> archive: PostgreSQL relational data + Fernet-encrypted settings +
> KV applicative state.
>
> **Out of scope.** Off-site replication setup, cron schedule design,
> incident triage / escalation flow — those live (or will live) in
> `RUNBOOK-incidents.md` (TODO Batch 3, see §10).

---

## 1. Why this runbook exists

`POST /api/backup/restore` (and its `upload-restore*` variants) only
restores the **applicative key/value layer**: `settings`,
`preferences`, `scheduler`, `watchlist`. It deliberately ignores
`pg_dump.sql` and never writes the embedded Fernet key on disk.

In a real incident (database corruption, full host wipe, fresh
install), that endpoint alone is **not enough**: the relational tables
must be re-created from the SQL dump *first*, and the Fernet key must
be put back in place *before* the application boots — otherwise the
encrypted secrets stay unreadable forever.

This runbook fills that gap with a single, reproducible, end-to-end
procedure that has been validated against a real production ZIP
(Batch 2C drill, 2026-05-02).

---

## 2. Anatomy of a backup ZIP

A backup ZIP produced by the current code (`manifest.version = "1.1"`)
contains:

```
mediakeeper_backup_YYYYMMDD_HHMMSS[_<label>].zip
├── manifest.json              metadata + components + encryption_key block
├── settings.json              applicative KV (sensitive values stay Fernet-encrypted)
├── preferences.json           per-user UI preferences
├── scheduler.json             scheduler tasks (replayed as UPDATE only, not CREATE)
├── watchlist.json             scanner snapshots
├── pg_dump.sql                full SQL dump (--clean --if-exists --no-owner --no-acl)
├── secrets/.encryption_key    active Fernet key (44 bytes, base64) — when persistent
└── logs/*.txt                 optional log bundle
```

Inspect a ZIP without extracting it:

```bash
unzip -l mediakeeper_backup_YYYYMMDD_HHMMSS.zip
```

Read the manifest only (no key extraction):

```bash
unzip -p mediakeeper_backup_YYYYMMDD_HHMMSS.zip manifest.json | jq .
```

Look for:

- `version` → must be `"1.1"` for this runbook.
- `components.pg_dump` → must be `true` if you expect database data.
- `encryption_key.status` → `"included"` or `"absent"`.

## 3. Pre-requisites

Before starting the restore, confirm you have:

- **Operator host** with: `unzip`, an `ssh` / `scp` client, and a
  container runtime (Docker Engine or compatible) **OR** a native
  PostgreSQL 16 client (`psql` ≥ 16). The drill that produced this
  runbook was run from Windows 11 + Docker Desktop on WSL2.
- **Read access to the backup source** — typically read-only SSH onto
  the NAS pointing at `BACKUP_PATH`. **Never rely on the NAS being
  writable**; copy the ZIP off it before doing anything.
- **The Fernet key** — embedded in the ZIP under
  `secrets/.encryption_key`, **or** retrievable from the original
  host's `/data/.encryption_key`, **or** held in
  `MEDIAKEEPER_ENCRYPTION_KEY` of the original deployment. Without it,
  encrypted settings will read back as empty strings after restore and
  silently break every external integration.
- **Database password** for the *target* host. The cluster password
  lives in `/data/.pg_password`, generated by `entrypoint.sh` on first
  boot. The dump itself does not carry it (and that's intentional —
  you're restoring data, not reconfiguring authentication).
- **A staging area** that is gitignored and isolated from production.
  In this project, the convention is `drill/` at the repository root
  (already in `.gitignore` since `b0ded2c`).

> ⚠️ **Treat the ZIP as a secret.** It contains the active Fernet
> key. Anyone with the ZIP can decrypt every API key, webhook,
> password and OAuth token stored in `settings`. Do not commit it,
> do not push it to a chat platform, do not paste its contents.

## 4. Procedure — drill restore on an isolated container

The procedure below is the exact sequence used during the Batch 2C
drill. It restores the ZIP into a **disposable** PostgreSQL container
and verifies that the data + Fernet key are intact, **without
touching the production database**.

### 4.1 Identify and copy the source ZIP

From the operator host, list candidates on the NAS (read-only):

```bash
ssh -p <port> <user>@<nas> "ls -lt <BACKUP_PATH>/*.zip | head -5"
```

Pick the most recent one whose filename matches
`mediakeeper_backup_YYYYMMDD_HHMMSS*.zip`, then copy it locally:

```bash
scp -O -P <port> <user>@<nas>:<BACKUP_PATH>/<filename>.zip ./drill/
```

The `-O` flag forces the legacy SCP transfer. It is required when the
NAS does not expose an SFTP subsystem (default on stock Synology
DSM); without it, `scp` fails with `subsystem request failed on
channel 0`.

Verify the file size matches what you saw on the NAS — that's a
cheap integrity check.

### 4.2 Extract locally (inside the staging area)

```bash
unzip -q ./drill/<filename>.zip -d ./drill/<filename>/
```

Then read the manifest and confirm:

```bash
cat ./drill/<filename>/manifest.json | jq .
```

- `encryption_key.status` should be `"included"`.
- `secrets/.encryption_key` should exist and be exactly **44 bytes**.

### 4.3 Spin up an ephemeral PostgreSQL container

```bash
docker run --rm -d --name mk-drill-pg \
  -e POSTGRES_PASSWORD=drill \
  -p 55432:5432 \
  postgres:16
```

- `--rm` ensures the container disappears on stop.
- No `-v` flag → no persistent volume; the database vanishes with
  the container.
- Port `55432` is deliberately non-default to avoid colliding with
  any local instance on `5432`.

If you need to reproduce the production locale exactly (the bundled
container defaults to `en_US.utf8` whereas production uses `C`),
add `-e LANG=C -e LC_ALL=C`. The drill ran without it because the
restore does not touch the cluster locale; the application code
relies on explicit `LOWER()` casts so it is locale-tolerant.

Wait until the cluster is ready:

```bash
for i in $(seq 1 30); do
  docker exec mk-drill-pg pg_isready -h localhost -U postgres -q && break
  sleep 1
done
```

### 4.4 Create the target database

```bash
docker exec mk-drill-pg createdb -U postgres mediakeeper_drill
```

Use a name that is **not** `mediakeeper_db` — that nominal mismatch is
your last line of defence against accidentally replaying a dump on
the production database.

### 4.5 Replay `pg_dump.sql`

```bash
docker cp ./drill/<filename>/pg_dump.sql mk-drill-pg:/tmp/pg_dump.sql
docker exec mk-drill-pg psql -U postgres -d mediakeeper_drill \
  -f /tmp/pg_dump.sql > drill_psql.out 2> drill_psql.err
echo "exit=$?  stderr lines=$(wc -l < drill_psql.err)"
```

Expected outcome:

- exit code `0`
- `drill_psql.err` empty (zero lines)
- `drill_psql.out` ends with a series of `ALTER TABLE` statements
  setting up constraints and indexes

If the stderr is non-empty, **stop the procedure** and read the
errors before going further. The dump uses `--clean --if-exists`
so it tolerates a fresh database, but a corrupted or truncated SQL
file will surface here.

### 4.6 Spot-check row counts

```bash
docker exec mk-drill-pg psql -U postgres -d mediakeeper_drill -c "
SELECT 'alembic_version' AS table_name, version_num::text AS info, NULL::bigint AS row_count FROM alembic_version
UNION ALL SELECT 'users',           NULL, count(*)::bigint FROM users
UNION ALL SELECT 'settings',        NULL, count(*)::bigint FROM settings
UNION ALL SELECT 'scheduler_tasks', NULL, count(*)::bigint FROM scheduler_tasks;
"
```

Compare against the values you expect from the source instance.
What this check actually validates:

- `alembic_version` matches the migration head the source was on
  when the dump was taken — that's the cheapest way to detect a
  schema-vs-code skew before you restore on a fresh deployment.
- `users`, `settings`, `scheduler_tasks` are non-zero on a populated
  source — zero values mean the dump replayed an empty schema, which
  is rarely what you want.

### 4.7 Verify the Fernet key

The key from the ZIP must successfully decrypt the encrypted
settings restored into the database. The script below does that
without ever printing or persisting the cleartext value.

```python
# drill/fernet_check.py — disposable, deleted by §4.8.
from pathlib import Path
from cryptography.fernet import Fernet, InvalidToken
import psycopg2

KEY_FILE = Path("/d/secrets/.encryption_key")  # mounted read-only
PREFIX = "fernet:v1:"

key = KEY_FILE.read_text(encoding="ascii").strip().encode("ascii")
f = Fernet(key)

conn = psycopg2.connect(
    host="host.docker.internal", port=55432,
    user="postgres", password="drill", dbname="mediakeeper_drill",
)
cur = conn.cursor()
cur.execute(
    "SELECT key, value FROM settings "
    "WHERE (key LIKE '%.api_key' OR key LIKE '%.token' "
    "    OR key LIKE '%.password' OR key LIKE '%.secret' "
    "    OR key LIKE '%.webhook_url') "
    "  AND value LIKE 'fernet:v1:%' "
    "ORDER BY key LIMIT 2"
)
ok = ko = 0
for k, v in cur.fetchall():
    try:
        plain = f.decrypt(v[len(PREFIX):].encode("ascii")).decode("utf-8")
        print(f"[OK]  {k}: ***{len(plain)} chars***")
        del plain
        ok += 1
    except InvalidToken:
        print(f"[KO]  {k}: InvalidToken — key mismatch")
        ko += 1
print(f"[SUMMARY] OK={ok} KO={ko}")
cur.close(); conn.close()
```

Run it inside a disposable Python container so cryptography +
psycopg2 leave no trace on the operator host:

```bash
docker run --rm \
  --add-host=host.docker.internal:host-gateway \
  -v "$(pwd)/drill/<filename>:/d:ro" \
  -v "$(pwd)/drill/fernet_check.py:/fernet_check.py:ro" \
  python:3.12-slim \
  sh -c "pip install --quiet cryptography psycopg2-binary && python /fernet_check.py"
```

A successful drill prints `[OK]` lines for the sample settings and
ends with `[SUMMARY] OK=2 KO=0`. The cleartext values must **never**
appear in the output, in the rendered logs, or in any report — only
the masked length (`***N chars***`).

### 4.8 Total cleanup

When the drill is done, remove every trace from the operator host:

```bash
docker stop mk-drill-pg                       # --rm wipes the container
rm -rf ./drill/<filename>/                    # extracted SQL + key
rm -f  ./drill/fernet_check.py                # ad-hoc script
rm -f  ./drill/drill_psql.{out,err}           # psql output
rm -f  ./drill/<filename>.zip                 # source archive
ls -la ./drill/                               # must show nothing but . and ..
```

Cleaning the source ZIP is not optional: it carries the active
Fernet key, and the NAS still has the original copy if you need to
re-run the drill.

## 5. Real recovery (production-bound)

The drill above runs against a disposable container so you can
practise without risk. The **real** recovery into a freshly bootstrapped
MediaKeeper deployment differs in two important ways:

1. **The Fernet key must land before the application boots.**
   `entrypoint.sh` will generate a brand-new key on first launch if
   `/data/.encryption_key` is missing. That new key cannot decrypt
   anything written under the previous one, so the restore must
   pre-populate the file:

   ```bash
   # On the target host, BEFORE the first `docker compose up -d`:
   unzip -p mediakeeper_backup_YYYYMMDD_HHMMSS.zip secrets/.encryption_key \
     | install -m 600 -o <puid> -g <pgid> /dev/stdin /<data-volume>/.encryption_key
   ```

   Equivalently you can set `MEDIAKEEPER_ENCRYPTION_KEY=<value>` in
   the deployment env. Either way, the key must already be there when
   `_load_fernet()` runs for the first time.

2. **The SQL dump replaces the bootstrap schema.** Drop in the SQL
   dump *after* PostgreSQL is up but *before* (or instead of) the
   first `alembic upgrade head` if you want a byte-perfect restore.
   The simplest reliable sequence on the production container is:

   ```bash
   # Container running, application stopped (or in a one-shot mode).
   docker exec mediakeeper psql -U mediakeeper -d mediakeeper_db \
     -f /restore/pg_dump.sql
   ```

   The dump already carries its own `alembic_version` row, so the
   subsequent `alembic upgrade head` at the next normal boot is a
   no-op when the dump matches the current code, and applies missing
   migrations otherwise.

The application-side restore (`POST /api/backup/restore`) is then
**optional**: it will replay `settings.json`, `preferences.json`,
`scheduler.json`, `watchlist.json` against the database, but the
SQL dump already contains those rows, so this step is redundant
after a full SQL restore. It is mostly useful when you want to
re-apply a *partial* backup (settings-only, etc.) on top of an
otherwise healthy database.

## 6. Post-restore validation checklist

Run through this list after a real recovery, before exposing the
service to users:

- [ ] `alembic_version` matches the migration head expected by the
      currently deployed code.
- [ ] Sample row counts (`users`, `settings`, `scheduler_tasks`)
      are non-zero and consistent with the source instance.
- [ ] At least one Fernet-encrypted setting (an `.api_key` or
      `.webhook_url`) decrypts cleanly through the application.
      A simple smoke test: open the admin settings page and confirm
      the masked secret reads back populated, not empty.
- [ ] Outbound integrations (Emby, OpenSubtitles, Discord webhooks,
      OAuth providers) succeed on a test call — that's the canonical
      proof that the Fernet key is right.
- [ ] Scheduled tasks visible in the admin UI list match what the
      source instance had.
- [ ] No errors of the form
      `[ENCRYPTION] Failed to decrypt a value` in the application
      logs after the first few minutes of uptime.

## 7. Troubleshooting

### `scp: subsystem request failed on channel 0`

OpenSSH ≥ 9.0 defaults to SFTP for `scp`, which a stock Synology
sshd does not advertise. Add `-O` (capital O) to force the legacy
SCP transfer:

```bash
scp -O -P <port> <user>@<nas>:<path> .
```

Alternatively, enable SFTP in DSM (Control Panel → File Services →
FTP → SFTP).

### `pg_dump.sql` replay reports schema conflicts

The dump uses `--clean --if-exists`, so it expects the target
database to already exist (which `createdb` covers) but it does
**not** create the database itself. If you see `ERROR: database
"mediakeeper_db" does not exist`, run `createdb -U postgres
mediakeeper_db` first.

If the dump comes from a deployment whose Alembic head is *newer*
than the code currently deployed on the target, the SQL will
introduce columns or tables the application does not know about.
The application will still start, but the audit cluster (`13-db-
schema-data`) recommends keeping deploys aligned: if you must
proceed, run the migrations missing from the *new* code after the
restore.

### Encrypted settings come back as empty strings

`[ENCRYPTION] Failed to decrypt a value` in the logs means the
Fernet key the application is using does not match the key the
dump was encrypted with. Two common causes:

- The container booted before the key file was placed under
  `/data/.encryption_key`, so `entrypoint.sh` generated a fresh
  key and pinned it.
- The key was rotated on the source instance after the backup was
  taken.

Recovery: stop the application, replace `/data/.encryption_key`
with the value embedded in the ZIP under `secrets/.encryption_key`,
restart. Encrypted settings recover transparently without re-running
the dump.

### Container locale mismatch

The Postgres image used in the drill defaults to `en_US.utf8`,
whereas production uses `C`. The dump replays cleanly under either
locale because the application uses explicit `LOWER()` casts and
ICU-independent collations, but if you want a 100% production-faithful
drill add `-e LANG=C -e LC_ALL=C` to the `docker run` command.

### Table count differs from earlier audits

`SELECT count(*) FROM information_schema.tables WHERE table_schema =
'public'` includes `alembic_version`. Earlier audits sometimes report
table counts that exclude it; a one-row delta between sources is
typically the version table being counted differently, not real
schema drift. Compare actual table names rather than counts when in
doubt.

## 8. Operator-side reminders

### Deployment is not automatic

There is no continuous-deployment pipeline for MediaKeeper today.
After every merge that changes runtime code, the operator has to:

1. Sync the Windows working tree to the NAS (typically `robocopy`
   from the development host).
2. Rebuild the application image on the NAS
   (`docker compose build && docker compose up -d`).
3. Confirm the new code is actually live — for backup-related
   changes, a quick check is:

   ```bash
   ssh -p <port> <user>@<nas> "docker exec mediakeeper grep -c \
     '_embed_encryption_key' /app/backend/services/backup/create.py"
   ```

   A zero count means the rebuild did not pick up the latest code.

The Batch 2C drill caught this exact failure mode: PR #16 (Batch 2A,
`pg_dump=True` by default) had been merged on `main` without the NAS
container being rebuilt, so production backups still shipped as
metadata-only ZIPs (~268 KB) for several days. Until a CD pipeline
is in place, **operator vigilance after every merge is mandatory.**

### What this runbook does not cover

- **Choosing a target for off-site replication.** Frozen project
  decisions exist (`V1=A` USB-encrypted DSM, `V2=A` 30 days daily
  + 1/week × 6 months, `V3=A` 14:00 window, `V6=A` documented in
  this folder) but they only kick in post-v1.0. Until then,
  off-site is intentionally **not configured** — the database is
  considered disposable until the v1.0 wipe.
- **Incident triage and escalation.** Who decides to roll back,
  who is on call, how the user community is notified — all of
  that lives in `RUNBOOK-incidents.md` (TODO Batch 3, see §10).
- **Code-level auto-restore via the API.** The current restore
  endpoint deliberately does not replay `pg_dump.sql`. A future
  batch (post-v1.0) may revisit that decision; until then, the
  manual procedure documented here is the supported path.
